1 · Why this platform exists

A bank that adds up each borrower's expected loss in isolation systematically understates how much it can lose at once.

Every borrower has a probability of default (PD). But defaults are not independent — and independence is exactly what a simple sum of per-borrower numbers assumes. Borrowers:

  • share a local economy — a regional shock hits a whole geography together;
  • move money between each other — a transaction network creates real financial dependence;
  • cluster around an anchor — when one person funds a family or a small ring (якорный человек), their default cascades to everyone who depended on them.

The tail of the portfolio loss distribution — the number that determines economic capital — is driven by these correlations, not by the averages. This platform models the correlation explicitly, prices its cost into every risk metric, and surfaces the clusters and anchors where contagion concentrates.

The one-sentence value: it converts marginal PDs + a money-flow graph + geography into a joint-default model, then reports risk-adjusted metrics at borrower, cluster, and portfolio level — so the risk department can act on hidden correlation, not just headline averages.

2 · The big picture in one diagram

Data flows left → right. Each block is one responsibility, owned by one role, with a typed hand-off to the next.

DE raw data → ML PD per borrower → DS graph · geo · transfer clusters → DS copula (joint default) → RISK metrics · stress · ratings → API agent / reports

The three intellectual leaps, in order:

1 · PD → marginal risk

A gradient-boosting model turns borrower features into a calibrated default probability. This is the marginal — each borrower alone.

2 · Graph → correlation

Money flows and shared geography become a correlation structure: who moves together. Communities and anchors are detected here.

3 · Copula → joint risk

A copula stitches the marginals together with that correlation, producing the joint default distribution that every risk metric reads from.

4 · Module — Ingestion & data contract

Data Engineer loaders.py · data_generator.py · config.py

What it does. Loads persons and transactions from real files (or generates a realistic synthetic network), validates them, and guarantees the one invariant everything else relies on: person_id is a unique integer used as the positional array index in every numpy operation downstream.

The logic. ColumnMapping lets you point the loader at your real column names (including geo_longitude / geo_latitude). Validation rejects duplicate / null ids, clamps PDs into [0,1], accepts PD as a percent or a fraction, range-checks coordinates, and re-indexes ids to a dense 0..n-1 range so array math is safe.

from src.data_eng import load_persons, load_transactions, ColumnMapping
m = ColumnMapping(person_id="customer_id", model_pd="pd",
                  geo_longitude="lon", geo_latitude="lat")
persons = load_persons("persons.csv", mapping=m)      # validated + reindexed
tx      = load_transactions("transfers.csv", mapping=m)
Why it matters for risk: bad ids or mis-scaled PDs silently corrupt every number downstream. This layer is the gate — if it passes, the rest is trustworthy.

5 · Module — Probability of default (the marginal)

ML Engineer pd_model.py · structural_pd.py

What it does. IndividualPDModel trains a logistic or gradient-boosting classifier on borrower features to predict model_pd ∈ [0,1]. StructuralPDModel adds an independent, Merton-style structural estimate as a cross-check.

The logic. The ML PD is the data-driven marginal. The Merton PD treats each borrower like a tiny firm whose "assets" are a capitalised income proxy and whose "debt" is their obligations; default is when assets fall below debt. When the two PDs disagree, that divergence is an early-warning signal — the structural view sees stress the statistical model hasn't priced yet.

Merton: PD = Φ(−d₂), d₂ = [ln(V/D) + (r − σ²/2)·T] / (σ·√T)
from src.ml import IndividualPDModel
model = IndividualPDModel(feature_columns=FEATURES, model_type="gradient_boosting")
model.fit(persons, target_col="default")
persons["model_pd"] = model.predict_proba(persons)
# validation AUC is reported in the fit result (e.g. val_auc ≈ 0.81)
Production note: the synthetic PD is illustrative. Calibration (does a predicted 3% actually default 3% of the time?) is the first thing to validate on real data — see METHODOLOGY.md §12.

6 · Module — Graph, geo clusters & transfer communities

Data Scientist graph_features.py · geo_clusters.py · transfer_clusters.py

6.1 · Transaction graph

TransactionGraph builds a sparse (CSR) directed money-flow graph and derives per-node features — degree, clustering coefficient, connected components, and neighbour-PD aggregates (how risky are the people you transact with). Sparse storage is what lets this scale to millions of nodes.

6.2 · Geo clusters — "who shares a local economy"

GeoClusterer runs DBSCAN on (latitude, longitude) with a great-circle (haversine) metric. eps_km is your only granularity knob; clusters are arbitrary-sized (no fixed k). Geographic outliers become unique negative ids → treated as independent in the copula (correctly, they share no local economy).

from src.analytics import GeoClusterer, GeoClusterConfig
gc = GeoClusterer(GeoClusterConfig(eps_km=8.0, min_samples=5)).fit(persons)
persons = gc.assign(persons)        # adds geo_cluster_id

6.3 · Transfer communities — "who moves money together"

TransferClusterer runs weighted Louvain community detection on the money-flow graph: people who send/receive among themselves form a community (transfer_cluster_id). resolution ↑ ⇒ more, smaller communities. Singletons become unique negative ids → independent.

from src.analytics import TransferClusterer, TransferClusterConfig
tc = TransferClusterer(TransferClusterConfig(resolution=1.0, min_cluster_size=4))
tc.fit(persons, tx)
persons = tc.assign(persons)        # adds transfer_cluster_id + anchor columns
g = tc.subgraph(cluster_id=0)    # inspect one community's money flow
Two independent lenses on correlation: geography (DBSCAN) and money flow (Louvain). The copula can use both at once (next sections), so a pair who share a city and trade money is more correlated than a pair who share only one.

7 · Module — Copulas (marginals → joint default)

Data Scientist copula_model.py · factor_copula.py · multi_factor_copula.py

What a copula does. It is the mathematical object that combines individual PDs with a correlation structure to produce the joint default distribution. Same marginals + stronger correlation ⇒ a fatter loss tail ⇒ more economic capital. That is the entire point.

Which copula to use

CopulaUse whenScaleTail
MultiFactorCopulageo and transfer both drive correlation (the recommended cluster model)up to 10MGaussian
FactorCopulaone systematic factor; want fat tails via Student-tup to 10Mt (set student_t=True)
CopulaDefaultModel('clayton')≤ 20k names; want explicit lower-tail dependence≤ 20klower-tail

The factor model (how it scales)

Instead of storing an n×n correlation matrix (impossible at millions), each borrower's latent asset value is built from a few shared factors plus an idiosyncratic part:

Aᵢ = Σₖ βᵢₖ · Yk,f(i) + √(1 − Σₖ βᵢₖ²) · εᵢ , default if Aᵢ ≤ Φ⁻¹(PDᵢ)

Borrowers in the same geo cluster share that geo factor draw; borrowers in the same transfer community share that transfer factor draw. The implied correlation is exact and cheap:

corr(Aᵢ, Aⱼ) = Σₖ βᵢₖ·βⱼₖ · 1[share factor k] → share geo only: β²; share both: 2β²; share none: 0
from src.copula import MultiFactorCopula
factors = persons[["geo_cluster_id", "transfer_cluster_id"]].to_numpy()
mfc = MultiFactorCopula().fit(persons["model_pd"].to_numpy(),
                             factors, betas=[0.30, 0.30])  # equal weight
Constraint: Σₖ βᵢₖ² < 1 per borrower (the model raises if you violate it). Fewer / larger factors ⇒ stronger clustering ⇒ fatter tail — choose granularity deliberately and validate with a variance-inflation check (tutorials/03).
Gaussian factor copulas have no tail dependence. For deep-stress clustering (everyone defaulting together in the extreme), use FactorCopula(student_t=True, nu=6) or the Clayton copula for small books.

8 · Module — Anchors & contagion (якорный человек)

Data Scientist / Risk Analyst transfer_clusters.py · cluster_metrics.py

The pattern. In many communities one person is the source of funds — a parent funding a family, a hub funding a small ring. If that anchor defaults, the dependents lose their inflow and tend to default too. A per-borrower view never sees this; it is a property of the structure.

┌──► dependent anchor ─┼──► dependent anchor defaults ⇒ inflow stops ⇒ dependents fail └──► dependent

How an anchor is detected

For every member of a community the model scores three structural signals and blends them:

anchor_score = 0.45·(money-source dominance) + 0.35·(dependent fraction) + 0.20·(is articulation point)
  • Source dominance — share of the community's internal inflow this node sends.
  • Dependent fraction — how many members draw a majority of their inflow from this node.
  • Articulation point — removing this node disconnects the community (a structural single point of failure).

The top scorer above a threshold becomes the cluster's anchor; its dependents are flagged; and the community gets a fragility score ∈ [0,1] = anchor's inflow share × (1 − redundancy of alternative funding sources).

New columns on persons: is_anchor, depends_on_anchor, anchor_score, cluster_fragility.

Pricing the cascade — anchor-contagion uplift

ClusterRiskMetrics.anchor_contagion_table() answers the headline question: if the anchor defaults, how much does this cluster's expected loss rise? It computes each member's PD conditional on the anchor defaulting (from the copula's joint probabilities) and compares the conditional cluster loss to the unconditional one:

uplift_ratio = EL(cluster | anchor defaults) / EL(cluster) ≥ 1 ≈ 1 → diversified, safe ≫ 1 → fragile, cascade risk
from src.risk import ClusterRiskMetrics
crm = ClusterRiskMetrics(calc, persons)          # calc = the metric calculator (§9)
contagion = crm.anchor_contagion_table()         # ranked by uplift_ratio, worst first
worst = contagion.iloc[0]
# "if anchor X defaults, cluster Y loss rises 3.5x (EL 12k → 42k)"

9 · Module — The risk-metric engine

Risk Analyst risk_adjusted_metrics.py · metric_comparison.py · risk_metrics.py

What it does. RiskRatioCalculator is the heart of the risk output. It builds a loss-covariance object once, then computes the seven risk-adjusted metrics at any level: per borrower, per any segment (geo cluster, transfer community, city, archetype …), or whole portfolio.

The correctness rule that makes the numbers right (INV-6)

Expected loss, profit and capital are additive — you can sum them across borrowers. Variance is not. Under correlation, the variance of a segment's loss is the block-sum of the loss-covariance matrix over that segment, never an average of per-borrower ratios.

Var(Loss_S) = Σi∈S Σj∈S LossCov[i,j] LossCov[i,j] = (EADᵢ·LGDᵢ) · (P(Dᵢ∩Dⱼ) − PDᵢ·PDⱼ) · (EADⱼ·LGDⱼ)
Never compute a segment metric as per_borrower.groupby(col).mean(). That ignores the off-diagonal covariance — exactly the contagion you are trying to measure — and understates risk. Always use by_segment(col), which does the block-sum.
from src.risk import RiskRatioCalculator
ead  = persons["exposure_at_default"].to_numpy()   # or an income proxy
calc = RiskRatioCalculator(mfc, persons, exposures=ead, lgd=0.45)

pb   = calc.per_borrower()                       # O(n), scales to millions
geo  = calc.by_segment("geo_cluster_id")         # block-sum, correct under correlation
comm = calc.by_segment("transfer_cluster_id")
port = calc.metric("sortino_copula")            # one number, whole book
dvr  = calc.diversification_ratio()              # ≥ 1; near 1 = concentrated
Reading the output of by_segment: the grouping key is returned in a column literally named segment (not the input column name). So after geo = calc.by_segment("geo_cluster_id"), read the cluster id as geo["segment"].

Scale: it never builds a dense n×n matrix above the threshold

For ≤ 20k names the loss-covariance is materialised densely (fast). Above that, each segment's block is computed on demand from the copula's blockwise joint probabilities — only m×m memory, never n×n. Same answer either way (verified to 1e-9 in the test suite).

10 · Modules — Stress, ratings, regimes, profiles

Stress testing RISK

risk_metrics.py — re-run the portfolio under a PD multiplier and a correlation boost; reports stressed EL / VaR and the % change. The copula state is always restored by a context manager (never left mutated).

Rating engine RISK

rating_engine.py — maps PD to AAA … Default and builds a 1-year migration matrix via P(Δt)=exp(G·Δt) for a forward outlook.

Regime-aware copula DS

flexible_probs.py — kernel-weights historical scenarios by similarity to today's macro state, classifies the regime (calm / moderate / stress), and re-tunes copula dependence accordingly.

Customer profile + watchlist RISK

customer_profile.py — a per-borrower dossier: tier, top contagion neighbours, systemic importance (PD uplift this borrower causes in others), and a watchlist of critical / high-tier names.

11 · How the risk department operates it

Four standing workflows. Each is a few lines through the agent façade (RiskAgentAPI), which orchestrates the modules and returns JSON-safe results — ideal for dashboards, schedulers, or an AI agent.

Workflow A — daily early-warning scan

Question: who is cheap on a correlation-blind metric (RAROC) but expensive on the correlation-aware one (copula-Sortino)? Those names carry hidden contagion risk.

from src.agents import RiskAgentAPI
api = RiskAgentAPI(persons=persons, transactions=tx)
api.run_pipeline()
flags = api.flag_divergences(z_threshold=1.5)
print(flags.summary)        # borrowers where RAROC and sortino_copula diverge → investigate

Workflow B — cluster & anchor contagion review

Question: which communities will cascade if their anchor defaults?

api.run_cluster_analysis(eps_km=8.0, min_cluster_size=4,
                         beta_geo=0.30, beta_transfer=0.30)
print(api.fragile_clusters(top_n=10).summary)   # ranked by anchor-contagion uplift
print(api.anchors().summary)                    # the anchor people themselves
print(api.cluster_report(cluster_id=3).summary)  # drill into one community

Workflow C — stress test

Question: how bad does it get if PDs double and correlation rises?

res = api.run_stress(pd_multiplier=2.0, correlation_boost=0.2)
print(res.summary)        # stressed EL / VaR and % change vs base
print(api.regime_status().summary)   # is today calm / moderate / stress?

Workflow D — single-name & portfolio review

print(api.query_borrower(42).summary)          # full dossier for one borrower
print(api.top_risks(n=20).summary)             # worst names by composite risk
print(api.portfolio_summary().summary)         # EL, VaR, ES, diversification
print(api.segment_metrics("risk_archetype").summary)
Every RiskAgentAPI call returns an AgentResult with .ok, .data (JSON-safe), .summary (human text), .warnings, .error — so it drops straight into a report, a dashboard, or an agent loop without post-processing.

12 · The seven metrics, decoded

All seven are produced by by_segment / per_borrower at every level. Riskiness metrics (CoV) are always ≥ 0; performance ratios flip sign when profit < 0, so for pure riskiness ranking use CoV.

MetricFormula (intuition)What it tells the risk team
coefficient_of_variationσindep / EL loss volatility per unit of expected loss, ignoring correlation.
coefficient_of_variation_copulaσcopula / EL same, with correlation. The gap vs the line above is the correlation penalty for that segment.
rarocE[Profit] / Capital risk-adjusted return on capital — but correlation-blind; pairs with Sortino for the early-warning signal.
sharpe_indep(E[Profit] − rf·Revenue) / σindep classic risk-adjusted performance, independence assumption.
sortino_indep(E[Profit] − h·Capital) / σindep performance over hurdle, downside only, no correlation.
sortino_copula primary (E[Profit] − h·Capital) / σcopula the headline number. Performance over hurdle vs the correlation-aware downside. Use this to rank segments.
sortino_simulated(E[Profit] − h·Capital) / σsim same as above but the downside is from a Monte-Carlo draw (captures full tail shape, incl. Student-t / Clayton tails).

Plus the portfolio diagnostics:

  • diversification_ratio = Σσᵢ / σportfolio ≥ 1. Near 1 ⇒ a concentrated, undiversified book.
  • VaR / ES (risk_metrics.py) — the loss quantile and the expected loss beyond it; ES ≥ VaR ≥ ~EL always.

14 · Rules, assumptions & honest caveats

Invariants the platform enforces (don't fight them)

  • person_id is a unique integer — the positional index for all numpy ops.
  • PDs stay in [0,1]; correlation matrices are PSD with unit diagonal.
  • Segment variance is the block-sum of the loss-covariance (INV-6) — never a mean of per-borrower ratios.
  • Stress tests restore copula state via a context manager.
  • The test suite (python test_copula_framework.py → 41 tests) gates every change.
Modelling caveats for production (documented in METHODOLOGY.md §12):
  • LGD defaults to a flat 0.45 — replace with a real loss-given-default model.
  • EAD uses an income proxy when exposure isn't supplied — feed true exposure.
  • PD calibration on synthetic data is illustrative — validate calibration and back-test on real outcomes before relying on the absolute numbers.
  • Gaussian factor copulas have no tail dependence — for deep stress use the Student-t factor copula or Clayton.
The correlation, copula and metric machinery is verified correct; these are the inputs a risk team must own.
